# GPT Image 2 / Banana 图像生成页 UI 重设计方案

日期：2026-04-24
项目：ai-essay-editor
范围：先优化 GPT Image 2 图像生成/编辑页面的 UI 与交互呈现，再将同一套 UI 代码与交互结构复用到 Banana 图像生成对话框；不改变现有业务逻辑与后端能力。

## 1. 目标

将当前图像生成页面升级为更接近大厂 AI 产品的体验，整体风格以 ChatGPT 式极简为主，结合品牌绿色作为强调色。

本次改造必须满足：
- 保留现有功能与逻辑不变
- 保留文生图 / 图像编辑两种模式
- 保留尺寸与分辨率选择能力
- 保留图像上传能力
- 保留文本输入框
- 用户点击“生成图像/编辑图像”后，立即进入类似 ChatGPT 的全屏对话页面
- 进入对话页面后，底部输入框上方继续保留相同能力的参数按钮，允许继续修改模式、尺寸/分辨率、参考图
- GPT Image 2 改造完成后，同样的 UI 代码与交互结构复用到 Banana 图像生成对话框
- UI 复用应优先通过共享组件/共享布局完成，而不是复制两套页面代码

## 2. 最终交互模型

采用双阶段结构：

### 阶段一：首页工作台
首页不是传统复杂表单，而是高保真 AI 创作入口。

首页承担：
- 用户首次输入图像需求
- 选择生成模式（文生图 / 图像编辑）
- 选择尺寸和分辨率
- 上传参考图或待编辑图片
- 发起本次任务

首页不承担：
- 展示完整会话过程
- 承载多轮上下文输出

### 阶段二：全屏对话页
用户点击“开始生成”或“开始编辑”后，立即跳转到全屏 ChatGPT 风格会话页。

对话页承担：
- 展示当前会话消息流
- 展示生成结果
- 支持追问、继续生成、继续编辑
- 在底部输入区上方保留参数按钮条
- 允许用户继续修改模式、尺寸/分辨率、参考图，再发起下一轮请求

## 3. 信息架构

### 3.1 首页结构
首页采用单中心焦点布局，不再强调“工作台面板”的厚重感。

结构顺序：
1. 顶部轻量导航
   - 返回首页
   - GPT Image 2 标识
   - 积分 / 历史入口
2. 中心 Hero 区
   - 页面标题
   - 简短说明文案
3. 模式切换条
   - 文生图
   - 图像编辑
4. 参数操作条
   - 比例按钮
   - 分辨率按钮
   - 参考图上传按钮
5. 主输入区
   - 大文本输入框
6. 推荐提示词 chips
7. 主行动按钮
   - 开始生成 / 开始编辑

### 3.2 对话页结构
对话页继续复用现有 GPT Image 2 聊天页逻辑，但改造成更完整的 AI 图像对话体验。

结构顺序：
1. 顶部导航栏
   - 返回
   - GPT Image 2 标识
   - 积分 / 历史入口
2. 中部消息流
   - 用户消息
   - AI 输出
   - 图像结果渲染
3. 底部输入区
   - 参数按钮条
   - 文本输入框
   - 上传入口
   - 发送按钮

## 4. 参数设计

## 4.1 参数原则
用户进入页面后，应当拥有自行选择尺寸与分辨率的权力。

因此：
- 首页首屏直接提供尺寸/分辨率相关操作入口
- 不隐藏到二级抽屉后才可见
- 对话页继续保留同等能力
- 用户在任意一轮都可修改参数

## 4.2 参数控件形态
推荐使用大厂常见的轻量参数按钮（pill buttons），而不是传统重表单下拉。

首页：
- 使用横向按钮组 / 胶囊按钮
- 例如：`1:1`、`4:3`、`16:9`
- 分辨率单独为按钮或轻量选择器
- 上传图片也是同一层级的胶囊按钮

对话页：
- 在底部输入框上方放置参数按钮条
- 继续显示当前所选模式、比例/分辨率、参考图状态
- 示例：`图像编辑` `1:1 · 1024×1024` `参考图 2 张`

## 4.3 参数值约束
比例和分辨率都只允许固定选项，不允许自由输入。

原因：
- 降低错误输入风险
- 提升大厂产品的一致性体验
- 更符合当前业务逻辑的稳定性要求

## 5. 首页视觉层级

视觉气质：
- ChatGPT 式极简留白为主
- 品牌绿色只用于强调态，不铺满整个页面
- 弱化边框、弱化大面积说明区、减少卡片堆叠

首页重点调整：
- 删除当前大块“全屏图像工作台”式传统表单感
- 将中心输入框作为视觉焦点
- 将模式、尺寸/分辨率、上传图统一整理为输入前的轻量控制条
- 右侧厚重信息卡改为更轻的结果摘要或直接取消
- CTA 按钮固定为明确主行动

视觉优先级：
1. 主输入框
2. 模式切换
3. 参数按钮条
4. 上传入口
5. 推荐提示词
6. 次级说明文字

## 6. 对话页视觉层级

对话页目标是更像真正的 ChatGPT 图像对话产品，而不是普通聊天页里拼接图像控件。

重点调整：
- 保留现有消息流逻辑
- 强化底部 composer 的产品感
- 将模式、尺寸/分辨率、上传入口组织成输入框上方参数条
- 参数条视觉上与 composer 一体化，但层级低于文本输入框
- 用户消息与图片结果继续沿用现有渲染能力

视觉优先级：
1. 最新消息 / 图片结果
2. 输入框
3. 参数按钮条
4. 发送按钮
5. 历史入口与顶部信息

## 7. 跳转与状态传递

点击首页“开始生成 / 开始编辑”后：
- 立即跳转到全屏对话页
- 将首页已选模式、尺寸/分辨率、上传图片、输入内容作为本轮对话初始上下文
- 不改变现有后端请求结构与业务处理方式
- 仅调整前端发起方式与 UI 展示方式

进入聊天页后：
- 当前参数状态同步显示在底部参数条
- 用户修改参数后，下一轮请求按最新选择发起

## 8. 复用范围

本次设计先以 GPT Image 2 为落地点，但目标不是只改一个页面，而是沉淀一套可复用的图像生成 UI 框架。

复用要求：
- GPT Image 2 首页与对话页优先完成改造
- Banana 图像生成对话框随后接入同一套 UI 组件与布局体系
- 两者共享：首页骨架、参数按钮条、输入区结构、上传区样式、聊天页 composer 结构
- 两者差异只保留在模型标识、文案、参数映射与请求字段层

## 9. 不变项

本次设计明确不改：
- 后端工作流逻辑
- 现有 GPT Image 2 请求能力
- 现有 Banana 请求能力
- 现有图像生成/编辑业务流程
- 现有消息渲染主逻辑
- 现有会话历史机制

## 10. 建议实现策略

为保证风险可控，建议采用“UI 重组优先、逻辑复用最大化”的策略：

1. 先在 GPT Image 2 上完成 UI 重构与布局重排
2. 将首页发起动作与现有聊天页逻辑对接
3. 在聊天页底部 composer 上方增加统一参数条
4. 将首页与聊天页中可共享的图像生成 UI 抽为共享组件
5. 参数状态在首页与聊天页之间保持同步
6. Banana 图像生成对话框接入同一套共享组件，仅做最小模型差异适配
7. 所有新增交互都尽量复用现有状态与请求字段

## 11. 验收标准

改造完成后，应满足以下结果：
- 首页视觉明显更接近大厂 AI 产品
- 首页保留图像模式、尺寸/分辨率、上传、文本输入能力
- 点击生成/编辑后立即进入全屏对话页
- 对话页底部输入框上方保留同类参数按钮
- 用户可在对话页继续修改参数并发起下一轮生成/编辑
- 功能逻辑与结果输出不退化
- 移动端与桌面端都保持清晰、简洁、一致的层级
- 移动端必须做真机感检查：在 iPhone/Android 浏览器里连续发送 3 条短消息、1 条长消息、1 次图片生成请求后，页面仍应像聊天工具而不是移动端落地页；头部高度不压内容，消息间距和可读行高接近 ChatGPT，底部输入区在键盘弹起后位置稳定且不遮挡最近消息
