# 故障排查 Runbook

遇到线上异常时，先记录时间、影响范围、用户反馈和错误截图，再按本文步骤排查。不要把真实密钥写进文档或代码。

## 1. 站点打不开

1. 打开 `/health`，查看运维状态页是否可访问。
2. 请求 `/api/health`，确认是否返回 `status: ok`。
3. 检查首页 `/`、价格页 `/pricing`、隐私页 `/privacy`、条款页 `/terms`、退款页 `/refund-policy`。
4. 查看应用日志，确认是否存在构建失败、运行时错误或反向代理错误。
5. 检查环境变量是否存在，但不要输出真实值。

## 2. 后台打不开或数据为空

1. 打开 `/admin`，确认登录页是否正常展示。
2. 登录后刷新数据，查看页面是否出现中文错误提示。
3. 如果用户列表为空，检查 Supabase 用户表和后台查询接口。
4. 如果订单为空，检查订单表、支付回调日志和订单状态。
5. 不要修改后台鉴权逻辑来绕过问题。

## 3. 支付成功但权益未到账

1. 收集用户 ID、本站订单号、支付平台订单号和支付时间。
2. 检查支付回调是否到达服务器。
3. 检查订单状态是否为已支付。
4. 检查用户积分、会员状态和交易流水是否更新。
5. 如果需要人工补偿，先保留订单和日志证据，再按既有管理流程处理。

## 4. Sentry 或监控没有数据

1. 确认 `SENTRY_DSN` 或 `NEXT_PUBLIC_SENTRY_DSN` 已在对应环境配置。
2. 确认配置值不是占位符。
3. 确认没有把服务端密钥放进 `NEXT_PUBLIC_` 变量。
4. 检查 Sentry 项目的环境过滤和采样率。
5. 确认外部监控正在请求 `/api/health`，且告警联系人正确。

## 5. 回滚

1. 暂停继续发布，保留日志和错误页面截图。
2. 回退到上一个已验证版本。
3. 不要覆盖 `.env.production`，不要删除数据库或持久化数据。
4. 回滚后重新检查 `/api/health`、`/health`、`/`、`/pricing`、`/admin`。
